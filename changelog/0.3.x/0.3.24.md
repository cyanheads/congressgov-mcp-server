# 0.3.24 — 2026-05-31

## Added

- **`congressErrorContracts`** — shared `errors[]` array with `not_found`, `rate_limited`, `invalid_request`, and `upstream_error` reason codes; declared on all 10 tool definitions ([#32](https://github.com/cyanheads/congressgov-mcp-server/issues/32)). The service layer populates `data.reason` and `data.recovery.hint` on every raised error, making error responses machine-readable for agents that need to branch on error class.

## Fixed

- **Upstream error classification** — `classifyUpstreamError` in `congress-api-service.ts` now maps all non-2xx responses to classified domain errors: 400 → `invalid_request`, 403 → `invalid_request`, 5xx → `upstream_error` ([#34](https://github.com/cyanheads/congressgov-mcp-server/issues/34)). The internal `api.congress.gov` request URL is no longer echoed in error messages; each error carries `data.reason` + `data.recovery.hint`.
- **Calendar validation in `validateIsoDateTime`** — `tool-helpers.ts` now rejects impossible dates (Feb 30, month 13, hour 25) client-side via `Date` round-trip check before any upstream call ([#35](https://github.com/cyanheads/congressgov-mcp-server/issues/35)). Previously, shape-valid but calendar-invalid ISO date-times were forwarded to Congress.gov, which returned an opaque HTTP 400.

## Changed

- **`devcheck.config.json`** — added `@cyanheads/mcp-ts-core` to the outdated dependency allowlist; framework released 0.9.18 mid-work; adoption is deferred (Casey-directed). `bun.lock` updated with transitive dependency refresh.
